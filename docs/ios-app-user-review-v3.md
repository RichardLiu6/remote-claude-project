# iOS App v3 用户体验评价

## 总评：7.5/10（v2: 5.5 -> v3: 7.5，+2.0）

## 一句话感受

从"骨架不错但不敢用"升级到了"可以认真考虑日常使用"。v2 的三个致命伤（isConnected 假绿灯、中文输入没着落、session 断了没反馈）全部修好了，连接状态指示器让人有安全感，IME 输入方案比预期更优雅。但横屏体验还很粗糙，快捷键栏依然缺 NL 键，距离完全替代 Web 终端还差最后一步。

## 详细评分

### 1. 首次启动体验 — 7/10（v2: 7 -> v3: 7，->）

**变化：** 无实质变化。

Settings 页版本号从 2.0 更新到了 3.0，但首次启动的引导、测试连接按钮、Bonjour 自动发现都没有加。Session 列表、服务器地址显示、设置持久化这些 v2 已有的优点保持不变。

这不是批评——v3 的精力放在了修核心 bug 上，首次启动体验本来就不是 P0。但下一个版本应该考虑了。

**仍然希望：**
- 首次启动引导（开 Tailscale -> 跑脚本 -> 输入 IP）
- 设置页"Test Connection"按钮
- Bonjour/mDNS 或二维码配置

### 2. 终端可读性 — 7/10（v2: 7 -> v3: 7，->）

**变化：** 无实质变化。

SwiftTerm 渲染质量依然出色，三主题保持。动态字号适配（cols < 70 自动缩小）仍然缺失，这在 iPhone SE/mini 上是实际问题。v3 聚焦在连接和输入层面的修复，终端渲染这块没有动，可以理解。

**仍然希望：**
- 自动检测 cols，不足 70 时自动缩小字号
- 设置页字号滑块旁显示预估 cols 数
- pinch-to-zoom 手势调整字号

### 3. 打字体验 — 8/10（v2: 6 -> v3: 8，+2 ↑）

**v3 做了什么：** 这是本次升级最大的亮点之一。新增了 `IMETextField` 类——一个不可见的 UITextField 覆盖层，专门处理中文/日文/韩文 IME 输入。

**中文输入真的能用了吗？** 从代码设计来看，答案是肯定的。`IMETextField` 的方案非常聪明：

1. **diff-based 检测**（和 Web 端同源思路）：不监听 composition 中间态，而是在 `textDidChange` 中比较 `currentText` 和 `previousText`，只在 IME 确认提交后才发送差量文本。`isComposing` 通过 `markedTextRange != nil` 判断，这是 iOS IME 的标准信号——有候选拼音时 `markedTextRange` 非空，选词确认后变 nil。这个方案从原理上避免了重复发送的问题。

2. **backspace 处理**：diff 检测到字符减少时，按差值次数发送 `\u{7F}`（DEL）。还重写了 `deleteBackward()`，在 text field 已清空时仍能发送退格——这是 Web 端经常遇到的边界情况。

3. **特殊键处理**：通过 `keyCommands` 注册了方向键（上下左右）、Esc、Tab、14 个 Ctrl 组合键（a/b/c/d/e/f/k/l/n/p/r/u/w/z）。这比 v2 直接依赖 SwiftTerm 的键盘处理要全面得多——外接蓝牙键盘的场景也覆盖了。

4. **防溢出**：text field 累积到 100 字符后自动清空重置，避免内存泄漏。

**不好的地方：** 快捷键栏依然是 9 个按钮（Tab/^C/Esc/Up/Down/^D/^Z/^L），依然缺少 NL（换行/发送）按钮。这仍然是每次给 CC 发指令的高频操作。v3 在 `IMETextField` 里处理了 `textFieldShouldReturn` 发送 `\r`，所以系统键盘上的 Return 键能用——但问题在于 CC 的多行输入模式下，用户需要的是"发送换行但不提交"的 NL 键，而不是 Return。

左右方向键也依然缺失于快捷键栏（虽然 `IMETextField.keyCommands` 支持了外接键盘的方向键，但触屏软键盘上没有方向键按钮）。

**得分提升理由：** 中文输入从"完全未验证的隐患"变成了"架构正确、原理可靠的解决方案"。外接键盘支持从无到有（Ctrl 组合键 + 方向键 + Esc + Tab）。这两项加起来值 +2 分。

### 4. 滚动浏览 — 3/10（v2: 2 -> v3: 3，+1 ↑）

**变化：** 无直接代码变化，但 v2 评价时我已经注意到 `TerminalContainerView` 的触摸路由其实是接上的。v3 没有在滚动上做新工作。

+1 分的理由：v3 的 `onScrollModeChanged` 回调通过 coordinator 更新 SwiftUI 的 `inScrollMode` 状态，顶栏的 SCROLL 标签显示逻辑更清晰了。这至少在 UI 反馈层面改善了滚动体验的可感知性。

**仍然存在的问题：**
- 未经真机实测验证端到端的滚动流程
- 退出 copy-mode 仅靠点击，没有明显的浮动退出按钮
- 触摸事件被 `TerminalContainerView` 完全拦截，SwiftTerm 的点击定位光标功能失效
- 没有滚动位置指示器

**仍然希望：**
- 真机实测验证完整流程：手指滑动 -> tmux copy-mode -> 看历史 -> 点击退出
- 加浮动"Exit Scroll"按钮
- 滚动时显示半透明的行号/位置指示条

### 5. 复制粘贴 — 5/10（v2: 4 -> v3: 5，+1 ↑）

**变化：** 选择模式的触发路径更可靠了——长按检测有了明确的 500ms 延时 + 5px 移动阈值消歧（防止滚动误触发），触发时有 heavy 级别的触觉反馈。这让"长按进入选择"变成了一个有明确物理回馈的操作，而不是 v2 时代的"碰运气"。

overlay 关闭时自动复制选中文本 + success 触觉反馈的设计依然保持，体验连贯。

**仍然存在的问题：**
- 快捷键栏依然没有 Select 按钮，长按的可发现性仍然差
- overlay 默认 `selectAll(nil)` 全选整个 buffer，选中间一小段代码仍然繁琐
- 剪贴板桥接的"Paste"措辞仍有误导（实际是"发送到终端"）

**仍然希望：**
- 快捷键栏加 Select 按钮
- overlay 不默认全选，让用户自己拖选
- 加明确的"Copy"按钮（除了 Done）

### 6. 网络稳定性 — 9/10（v2: 7 -> v3: 9，+2 ↑）

**v3 做了什么：** 这是 v3 改进最深刻的一个维度。三个大改：

**1. ConnectionState 枚举替代 isConnected 布尔值。** v2 的 `isConnected` 在 `task.resume()` 后立即设为 true，造成假绿灯。v3 引入了四态枚举：`.disconnected` / `.connecting` / `.connected` / `.reconnecting(attempt:maxAttempts:)`。关键设计：`connectionState` 只在收到第一条 WebSocket 消息后才变为 `.connected`（`hasReceivedFirstMessage` 标记），彻底解决了假绿灯问题。每个状态都有对应的 `statusText` 和 `statusColor`，UI 层直接读取即可。

**2. Session 结束检测。** 新增 `DisconnectReason` 枚举（`.sessionEnded` / `.networkError` / `.serverUnreachable` / `.maxRetriesExceeded`），在 `handleDisconnect` 中根据错误码（URLError.cancelled/unknown、POSIX ECONNREFUSED/ECONNRESET/EPIPE）判断是 session 正常结束还是网络异常。session 结束时不自动重连，而是通过 `onSessionEnded` 回调弹出 Alert，让用户选择"Back to Sessions"或"Reconnect"。这个设计非常正确——之前 session 结束后 App 会疯狂重试 10 次然后报"连接失败"，用户完全不知道发生了什么。

**3. reconnect() 方法。** 新增手动重连入口，Alert 的"Reconnect"按钮直接调用。内部清理状态（重置 attempts、清除 sessionEndedByServer 标记、cancel 旧 task）后重新建立连接。

**连接状态指示对信任感的影响：** 巨大。v2 的绿色/红色二值指示器让人不敢信——进地铁时"绿灯闪一下变红"的体验比没有指示器更糟。v3 的四态指示器（绿点 + "Connected" / 黄点 + "Connecting..." / 黄点 + "Reconnecting (3/10)..." / 红点 + "Disconnected"）让我随时知道 App 在做什么。特别是 `reconnecting` 状态显示当前尝试次数和最大次数（比如"3/10"），这种透明度让等待变得可忍受——我知道它在努力，而不是卡死了。

**仍然缺失：**
- NWPathMonitor 网络状态监听（WiFi 切蜂窝时主动重连）
- `scenePhase` 变化时的重连检查（App 从后台切回前台）
- 断线时终端区域的半透明遮罩

**得分提升理由：** isConnected bug 是 v2 最严重的技术问题之一，v3 用 ConnectionState 枚举从根本上解决了。Session 结束检测让用户不再面对莫名其妙的"连接失败"。这两项加起来让网络层从"能用但有坑"变成了"可以信赖"。

### 7. 语音功能 — 6/10（v2: 6 -> v3: 6，->）

**变化：** 无实质变化。VoiceManager 的架构和功能保持不变。

v2 提到的问题（多段语音互相 cancel、无音量/语速控制、`lastSpokenText` 没有在 UI 展示）全部保持原样。可以理解——语音不是 v3 的重点。

**仍然希望：**
- 多段语音排队播放
- 语速控制（1x/1.5x/2x）
- 播放时顶栏显示文本摘要
- 耳机线控支持

### 8. 通知 — 6/10（v2: 6 -> v3: 6，->）

**变化：** 无实质变化。NotificationManager 保持原样。

固定通知文字、缺少分类分组、硬编码的匹配模式等问题依旧。同样可以理解——不是 v3 的优先级。

**仍然希望：**
- 通知内容包含 CC 输出的最后一行摘要
- UNNotificationCategory 支持"查看"action
- 模式匹配可配置化

### 10. 整体感受 — 7.5/10（v2: 5 -> v3: 7.5，+2.5 ↑）

**愿不愿意从 Web 切换到 App？**

可以开始尝试了。

v2 的三个核心卡点——"发指令时中文输入不敢用"、"看输出时不知道连没连上"、"session 结束了不知道怎么回事"——在 v3 中全部得到了实质性解决。作为一个每天在地铁上用手机操控 CC 的人，v3 给我的最大改变是**信任感**。

以前用 v2，我心里总在嘀咕："绿灯是真连上了吗？""拼音打完会不会发两遍？""CC 跑完了 session 关了怎么办？"这些不确定性让我宁愿用 Web 终端——虽然 Web 也不完美，但至少我踩过的坑都知道边界在哪。

v3 的 ConnectionState 四态枚举 + session 结束 Alert + IMETextField 三件套组合，把"不确定"变成了"透明"。我能清清楚楚看到连接状态，中文输入有了架构上正确的 diff-based 方案，session 结束时 App 会告诉我发生了什么并给出选择。这种透明感让我愿意把日常使用从 Web 迁过来试试。

**横屏体验如何？**

从代码看，v3 做了安全区域适配：`GeometryReader` 读取 `safeAreaInsets`，顶栏加了 `top` padding，快捷键栏加了 `bottom` padding，终端区域加了 `leading`/`trailing` padding。这意味着：

- **竖屏（刘海/灵动岛）：** 顶栏不会被刘海遮挡，底部 InputAccessoryBar 不会被 Home Indicator 盖住。
- **横屏：** 左右两侧有 safeAreaInsets padding，终端不会延伸到曲面屏边缘或灵动岛区域。

但实际横屏体验有几个问题：

1. **顶栏占用宝贵的垂直空间。** 横屏时屏幕高度本来就少，顶栏（session 名 + 状态 + 语音/剪贴板按钮）加上 InputAccessoryBar（44pt 高）会吃掉大约 80-90pt 的高度。在 iPhone 横屏时（可用高度约 375pt），终端实际可用面积不到 300pt，大约只能显示 15-18 行——勉强够用但很紧张。
2. **InputAccessoryBar 没有横屏优化。** 横屏时水平空间充裕，9 个按钮的 ScrollView 完全可以展开为单行，不需要滚动。但当前实现没有区分横竖屏布局。
3. **键盘弹出时更极端。** 横屏 + 系统键盘弹出后，终端可见区域可能只剩 5-8 行，几乎不可用。

**安全区域适配评价：** 代码层面做得正确——用 `GeometryReader.safeAreaInsets` 而不是硬编码数值，在所有 iPhone 型号（包括灵动岛、刘海屏、无刘海的 SE）上都能正确适配。但这只是"不出错"的基础，离"横屏好用"还有距离。

**Haptic 反馈评价：** 6 种场景的触觉反馈覆盖了关键交互节点：

| 场景 | 强度 | 体验评价 |
|------|------|---------|
| WebSocket 连接成功 | success (notification) | 恰到好处——连上了给个确认 |
| WebSocket 断开 | warning (notification) | 合理——提醒注意但不惊吓 |
| 快捷键栏按下 | light (impact) | 刚好——高频操作不能太重 |
| 终端 bell | medium (impact) | 正确——比声音提示低调但感知明确 |
| 长按进入选择模式 | heavy (impact) | 很好——明确告知"触发了特殊操作" |
| 复制文本成功 | success (notification) | 完美——"操作完成"的确认感 |

触觉反馈的强度层级设计得很讲究：light -> medium -> heavy 对应操作的重要性递增，notification 类的 success/warning 用于状态变化而非操作。这是 Web 终端永远无法提供的体验维度。

## Haptic 反馈补充说明

v2 只有 bell 震动一种触觉反馈。v3 增加到了 6 种，而且强度层级设计合理。这不是一个"有就行"的功能——触觉反馈的质量直接影响用户对 App 品质的感知。v3 在这方面做到了 iOS 平台的标准水准。

## 最想要的 5 个改进（v4 应该做什么）

### 1. 补齐快捷键栏的 NL 按钮和左右方向键
这是从 v2 延续到 v3 始终未解决的第一优先级。NL（换行但不提交）是给 CC 发多行指令时最常按的键，比 ^D/^Z/^L 的使用频率高 10 倍以上。左右方向键在编辑命令行时必不可少。建议将 InputAccessoryBar 扩展为两行，或者把低频的 ^D/^Z/^L 收进长按子菜单，把空间让给 NL/Left/Right/Select/Paste。

### 2. App 前后台切换时自动重连 + NWPathMonitor
v3 修好了连接状态指示，但仍然缺少主动重连机制。用手机的人每 5 分钟就会切一次 App（看微信、回消息），切回来发现断了要等被动重连，这个延迟在通勤场景下很痛苦。需要：(1) `scenePhase` 从 `.background` 变 `.active` 时立即检查连接并重连；(2) NWPathMonitor 监听网络路径变化（WiFi -> 蜂窝），在路径变化后主动触发重连。

### 3. 真机验证触摸滚动的端到端流程
滚动代码从 v1 就写好了，但到 v3 都没有经过真机实测验证。这个功能对日常使用至关重要——CC 的回复经常几十行，不能回看历史就等于瞎了一半。需要验证：手指滑动触发 tmux copy-mode -> 看到历史内容 -> 惯性滚动流畅 -> 点击退出 copy-mode -> 回到正常输入。如果有 bug，v4 必须修。

### 4. 横屏模式优化
当前的安全区域适配只是"不出错"，远未达到"好用"。v4 应该：(1) 横屏时自动隐藏或折叠顶栏（下拉手势展开）；(2) InputAccessoryBar 横屏时利用宽屏优势展开更多按钮；(3) 检测横屏 + 键盘弹出时终端可见行数，如果低于 10 行则自动缩小字号或隐藏非必要 UI。

### 5. Session 内快速切换
目前要切换 session 必须关闭当前终端 -> 返回 SessionPickerView -> 点击另一个 session。Web 端支持顶栏下拉切换，App 至少应该在顶栏加一个 session 列表入口，或者支持左右滑动切换。对于同时跑多个 CC session 的用户，这能节省大量操作。

## 会不会推荐给朋友？

可以有条件地推荐了。

如果朋友主要在竖屏下使用、打字以英文为主、不太需要回看长历史——那 v3 已经可以用了，而且连接状态指示和触觉反馈会让他觉得"比 Safari 里打开的网页终端高级"。

如果朋友是重度中文输入用户、经常需要回翻 CC 的长回复、会在横屏下工作——我会建议他再等等 v4。

从 v2 的"不推荐"到 v3 的"可以有条件推荐"，这是一个质的变化。v3 证明了一件事：修好核心 bug 比堆砌新功能更能提升用户信任。
