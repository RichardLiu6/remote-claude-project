# iOS App v4 用户体验评价

## 总评：8.5/10（v2: 5.5 -> v3: 7.5 -> v4: 8.5，+1.0）

## 一句话感受

从"可以认真考虑日常使用"升级到了"已经可以作为主力工具"。v4 终于解决了从 v2 就开始积压的第一优先级——NL 按钮和方向键——同时补上了网络感知和前后台重连这两个"信任感最后一公里"。启动缓存直接跳过 session 选择页，让"掏出手机就能用"变成了现实。这一版的主题词是"零摩擦"：零等待启动、零操作重连、零步骤切 session。

## 详细评分

### 1. 首次启动体验 — 8.5/10（v3: 7 -> v4: 8.5，+1.5 ↑）

**v4 做了什么：** 启动缓存（auto-connect to last session）。

这一个功能改变了整个启动体验。`SessionPickerView` 在 `loadSessions()` 完成后调用 `autoConnectIfPossible()`，检查 `UserDefaults` 中的 `last_session_name`，如果它还在活跃 session 列表中，直接 `selectedSession = session` 触发 `fullScreenCover` 进入终端。从用户视角：打开 App -> 看到一闪而过的 session 列表 -> 自动进入上次的 session -> 已经在终端里了。整个过程大约 1-2 秒，中间无需任何点击。

`TerminalView.onAppear` 中的 `UserDefaults.standard.set(sessionName, forKey: "last_session_name")` 确保每次进入新 session 都更新缓存，`switchToSession()` 中也同步更新——这意味着 session 切换后下次启动会连接最新的 session，而不是旧的。逻辑闭环，没有遗漏。

**为什么加 1.5 分：** 对于通勤场景来说，"打开 App 就能用"和"打开 App -> 等加载 -> 点 session"是截然不同的体验等级。前者意味着我可以在地铁门关上的 3 秒内看到 CC 的最新输出，后者意味着我要双手操作、等网络、找到对的 session、点击、再等连接。启动缓存消灭了最少 2 次点击和 3 秒等待。

**还有一个细节值得注意：** `hasAttemptedAutoConnect` 标记确保 auto-connect 只在首次加载时触发。如果用户手动返回 session 列表（比如点了 X 按钮），再次 `loadSessions()` 时不会自动跳进去——用户的"返回"意图被尊重了。这是一个小但重要的交互设计决策。

**仍然希望：**
- 首次启动引导（开 Tailscale -> 跑脚本 -> 输入 IP）
- 设置页"Test Connection"按钮
- auto-connect 时显示一个短暂的 toast（"Reconnecting to claude..."）而不是直接跳转，给用户一个心理预期

### 2. 终端可读性 — 7/10（v3: 7 -> v4: 7，->）

**变化：** 无实质变化。

SwiftTerm 渲染质量依然出色，三主题保持。动态字号适配（cols < 70 自动缩小）仍然缺失。v4 的精力放在了输入栏和网络层面，终端渲染没有动，连续三个版本未改。

不扣分是因为 v4 有更重要的事情要做（NL 按钮、网络感知），但 v5 应该认真考虑了——尤其是 iPhone SE/mini 的可读性问题。

**仍然希望：**
- 自动检测 cols，不足 70 时自动缩小字号
- 设置页字号滑块旁显示预估 cols 数
- pinch-to-zoom 手势调整字号

### 3. 打字体验 — 9/10（v3: 8 -> v4: 9，+1 ↑）

**v4 做了什么：** 快捷键栏从 9 键扩展到 16 键，NL 按钮放在第一位且视觉突出。

逐一分析 v4 新增的 7 个按键：

**NL 按钮（绿色，快捷栏第一位）：**

这是从 v2 就开始呼唤的第一优先级，终于落地了。`InputAccessoryBar` 的 `nlButton` 用了深绿底色 (`Color(red: 0.2, green: 0.45, blue: 0.3)`) + 白色 bold 字体 + 14px padding——比其他按钮明显突出，一眼就能找到。放在 `HStack` 的最左端，拇指最容易触及的位置。

NL 发送的是 `\r`（和 Return 键一样），这在 Claude Code 的多行输入场景下代表"确认发送当前输入"。有人可能会问：这和软键盘自带的 Return 有什么区别？区别在于：(1) 软键盘的 Return 键在打字时拇指要从屏幕底部跨越到键盘区域中间位置，NL 在快捷栏最左端，位置固定且靠下，盲按命中率更高；(2) 输入中文时 IME 候选区域可能遮挡 Return 键，NL 不受影响；(3) NL 的绿色视觉锚点让"输入完毕 -> 发送"这个高频动作有了固定的肌肉记忆位置。

**NL 解决了多少打字痛苦？** 至少 60% 的快捷键栏使用场景。在 CC 的对话模式中，每一轮指令的最后一步都是"按 NL 发送"。如果一天给 CC 发 50 条指令，就是 50 次 NL。v3 的 9 键栏没有 NL，用户要么去按软键盘的 Return（位置不固定、可能被 IME 遮挡），要么根本不知道怎么发送。NL 按钮把这个高频动作从"需要找"变成了"闭着眼睛按"。

**左右方向键（arrow.left / arrow.right）：**

终于补上了。编辑命令行时左右移动光标是基本操作，之前只能靠外接键盘的方向键（`keyCommands`）或者重新输入。现在和上下方向键一起放在分隔线后面的方向键区域，四个方向齐全，用 SF Symbol 图标，视觉清晰。

**Shift-Tab（`\u{1B}[Z`）：**

用于反向补全。在 Claude Code 中用得不多，但在 shell 的 tab-complete 场景下偶尔需要。属于"有了不嫌多"的按键。

**^A（行首）、^E（行尾）、^R（反向搜索）：**

三个 readline 快捷键。^A 和 ^E 在编辑长命令时使用频率极高——比如 CC 生成了一条 git 命令，你想在行首加个 `echo ` 测试一下，^A 一按光标就到行首了。^R 的反向搜索在找历史命令时很有用。这三个键的加入让快捷栏覆盖了 readline 的核心操作。

**快捷栏布局的改进：** 用 `Divider` 把 16 个按键分成了三个逻辑组：(1) NL + 常用键（Tab/^C/Esc）；(2) 方向键（四个箭头）；(3) Ctrl 组合键（^A/^E/^R/^D/^Z/^L）。视觉上的分组让按键更容易定位，不需要在一排按钮里数位置。

**外接键盘检测：** `setupExternalKeyboardDetection()` 通过监听 `keyboardWillShowNotification` 的 `keyboardFrameEndUserInfoKey` 判断键盘高度——外接键盘时高度 < 100pt（只有候选条），软键盘时 > 200pt。检测到外接键盘后 `hasExternalKeyboard = true`，`InputAccessoryBar` 的 `isHidden` 为 true 就不渲染了。这很聪明——外接键盘已经有物理方向键和 Ctrl 组合键了，快捷栏完全多余，还白白占 44pt 高度。

**得分提升理由：** NL 按钮单独值 +0.5 分，其他 6 个新按键 + 外接键盘检测值 +0.5 分。v4 的快捷栏终于达到了 Web 端快捷栏的功能覆盖度，同时还多了 Shift-Tab 这种 Web 端没有的按键。打字体验从"能用"进入了"顺手"的阶段。

**仍然希望：**
- 快捷栏加 Select/Paste 按钮（目前 Select 靠长按触发，可发现性差）
- 低频的 ^D/^Z/^L 考虑长按子菜单收纳，给高频键更多空间
- NL 按钮考虑加一个极短的 0.1s 颜色脉冲反馈（按下时变亮绿），加强"已发送"的感知

### 4. 滚动浏览 — 3/10（v3: 3 -> v4: 3，->）

**变化：** 无直接代码变化。

`ScrollGestureHandler` 代码和 v3 完全一致。触摸滚动的端到端流程仍然没有经过真机实测验证。连续三个版本没有在这个维度做工作。

这开始变成一个问题了。v4 在"输入"和"网络"上做了大量改进，但"输出"端的核心能力——回看历史——始终处于"代码写好了但没验证"的状态。在 CC 的长回复越来越常见的今天（动辄几十行代码 + 解释），不能可靠地回滚浏览是一个实际的日常痛点。

**仍然希望：**
- 真机实测验证完整流程：手指滑动 -> tmux copy-mode -> 看历史 -> 点击退出
- 加浮动"Exit Scroll"按钮
- 滚动时显示半透明的行号/位置指示条

### 5. 复制粘贴 — 5/10（v3: 5 -> v4: 5，->）

**变化：** 无实质变化。

长按选择的触觉反馈和 overlay 机制保持不变。快捷键栏新增了 16 个按键但没有 Select 按钮。可发现性问题依旧——新用户不知道长按可以选择文本。

**仍然希望：**
- 快捷键栏加 Select 按钮
- overlay 不默认 `selectAll(nil)` 全选
- 加明确的"Copy"按钮（除了 Done）

### 6. 网络稳定性 — 9.5/10（v3: 9 -> v4: 9.5，+0.5 ↑）

**v4 做了什么：** 两个大改——NWPathMonitor 网络感知 + scenePhase 前后台重连。

**1. NWPathMonitor 网络感知。**

`NetworkMonitor` 是一个全新的类，封装了 `NWPathMonitor` 的完整生命周期。关键设计：

- **三种网络类型检测：** WiFi / Cellular / Ethernet，通过 `NWPath.usesInterfaceType` 判断。UI 层的 `connectionType` 可以告诉用户当前是什么网络。
- **三种状态转换检测：** (a) `!wasConnected -> nowConnected` = 网络恢复，触发 `onNetworkRestored`；(b) `wasConnected -> !nowConnected` = 网络丢失，触发 `onNetworkLost`；(c) `wasConnected && nowConnected && type changed` = 网络类型切换（WiFi -> Cellular），也触发 `onNetworkRestored`。第三种情况特别重要——WiFi 切蜂窝时底层 TCP socket 通常已经断裂，但 URLSessionWebSocketTask 可能还没检测到（要等下一次 receive timeout），`NetworkMonitor` 能提前触发重连。
- **NO NET 红色标签：** 当 `!networkMonitor.isConnected` 时，顶栏显示一个红底白字的 `NO NET` 标签（9pt bold monospaced，4px horizontal padding）。这个标签和连接状态点是互补的：连接状态点告诉你"WebSocket 的状态"，NO NET 标签告诉你"底层网络的状态"。两者同时显示时（比如网络刚断、WebSocket 还没 timeout），用户能清楚看到"网络没了，WebSocket 还在挣扎"。

**网络感知对信任的提升有多大？** 非常大。v3 的四态连接指示器解决了"WebSocket 连上了吗"的信任问题，但 v4 的 NO NET 标签解决了一个更基础的信任问题："我的网络通吗？"。通勤场景下进出地铁、电梯、地下停车场，网络频繁断续。v3 时代，用户看到黄色"Reconnecting..."但不知道是网络问题还是服务器问题，只能干等。v4 时代，看到 NO NET 就知道"网没了，等信号恢复就好"，看到没有 NO NET 但在 Reconnecting 就知道"网络没问题，是服务器的事"。这种诊断能力让等待从"焦虑"变成"理解"。

**2. scenePhase 前后台重连。**

`TerminalView` 通过 `.onChange(of: scenePhase)` 监听 App 前后台切换。当 `newPhase == .active` 时，如果 `!wsManager.connectionState.isConnected`，立即调用 `wsManager.reconnect()`。

这解决了 v3 评价中提到的一个核心痛点："用手机的人每 5 分钟就会切一次 App（看微信、回消息），切回来发现断了要等被动重连。"v4 把被动等待变成了主动检查。从用户体验来说：从微信切回 Claude Terminal -> App 检测到 disconnected -> 立即 reconnect -> 1-2 秒后看到绿灯 Connected。比 v3 的"切回来 -> 发现黄灯 Reconnecting -> 等 exponential backoff -> 也许 5-10 秒后才连上"快了一个量级。

**两个机制的协同效果：** NWPathMonitor 在 App 后台运行时也在监听网络变化。当用户在地铁里把 App 切到后台，网络断了又恢复了，NWPathMonitor 触发 `onNetworkRestored` -> `wsManager.reconnect()`。然后用户切回前台时，WebSocket 可能已经重连好了。这种"后台悄悄修好"的体验是最高级的——用户甚至感觉不到断过线。

**得分提升理由：** NWPathMonitor + scenePhase 重连 + NO NET 标签把 v3 的网络层从"被动响应"升级到了"主动感知"。v3 已经是 9 分了（解决了假绿灯和 session 结束检测这两个核心问题），v4 在此基础上加了最后半分——从"可以信赖"到"几乎无感知"。不给满分是因为还缺断线时终端区域的半透明遮罩（告诉用户"当前看到的内容可能不是最新的"）。

**仍然缺失：**
- 断线时终端区域半透明遮罩
- NWPathMonitor 的网络类型变化（WiFi -> Cellular）在 UI 上没有显示具体类型，只有 NO NET 标签显示"没网"

### 10. 整体感受 — 8.5/10（v3: 7.5 -> v4: 8.5，+1 ↑）

**零操作启动体验有多好？**

打开 App -> 自动连接 last session -> 看到终端 -> NL 绿色按钮就在左下角。从解锁 iPhone 到"可以给 CC 发指令"的时间：大约 3 秒。v3 需要：打开 App -> 等加载 session 列表 -> 点 session -> 等 WebSocket 连接 -> 找到 Return 键，大约 8-10 秒 + 2 次点击。

这 5-7 秒的差距在通勤场景下是质变。等地铁的 30 秒间隙、上电梯前的 10 秒、等红灯的 15 秒——这些碎片时间以前不够"打开 App + 选 session + 等连接"，现在够了。App 变成了一个"随时可以瞟一眼"的工具，而不是"需要专门腾出时间来使用"的工具。

**NL 按钮对日常工作流的影响：**

最直观的变化：给 CC 发指令的完整操作从"打字 -> 在键盘区找 Return -> 按下"变成了"打字 -> 按绿色 NL"。绿色视觉锚点 + 固定最左位置 = 用了两三次就能盲按。NL 的存在还改变了心理预期——以前看到快捷栏会想"嗯这些都是辅助键"，现在看到绿色 NL 会想"这就是发送键"，操作意图更明确了。

**网络感知 + 重连对信任的提升：**

v3 让我"信任连接状态指示器"，v4 让我"忘记网络这件事"。最好的用户体验是让用户不需要思考的体验——v4 的 NWPathMonitor + scenePhase 重连做到了这一点。进地铁信号断了？看到 NO NET 标签就知道了，不用猜。出地铁信号恢复？App 自动重连了，不用手动操作。从微信切回来？已经连好了。网络层变成了一个"我不需要管它，它自己会搞定"的东西。

**Session 切换体验如何？**

点 session 名 -> 弹半屏 sheet -> 看到所有 session 列表 + 当前标记（绿色 checkmark）-> 点另一个 -> 自动断开旧连接、建立新连接。整个过程不需要离开终端视图，不需要先"关闭" -> "返回" -> "重选"。sheet 用了 `.presentationDetents([.medium])` 只占半屏，底下的终端还隐约可见，不会产生"我离开了终端"的不安感。

`SessionSwitcherSheet` 的 UI 细节做得不错：当前 session 有绿色 terminal 图标 + bold 字体 + 绿色 checkmark + 深绿背景，其他 session 是灰色图标 + medium 字体 + 蓝灰背景。视觉区分非常清晰。每个 session 还显示了 windows 数量和创建时间的相对描述（"2 hours ago"），信息密度恰到好处。

**愿不愿意从 Web 切换到 App？**

已经切换了。

v4 的 App 在我最在意的三个场景上全面超越 Web 端：

1. **掏出手机快速看一眼 CC 状态：** App 秒开 + auto-connect + 原生渲染 >> 打开 Safari + 等 CDN + 手动连接
2. **给 CC 发一条指令：** NL 绿色按钮 + 16 键快捷栏 >> Web 的 12 键栏 + 没有视觉突出的 NL
3. **地铁/电梯里断续使用：** NWPathMonitor + scenePhase 重连 + NO NET 标签 >> Web 端无网络感知

Web 端仍然赢的场景是长时间坐在桌前、需要频繁回看长输出、需要上传文件的时候。但这些场景下我通常已经在电脑前了，直接用桌面终端更高效。App 的主战场是移动场景，而 v4 在移动场景上已经是更好的选择。

## v4 新增功能深度评价

### NL 按钮解决了多少打字痛苦？

**量化估算：** 假设每天使用 App 给 CC 发 30-50 条指令，每条指令结尾都需要按一次"发送"。v3 时代这 30-50 次按键需要在软键盘区域找 Return 键（位置因键盘状态变化、被 IME 候选条遮挡、横竖屏位置不同），命中率大约 80%，每次按键耗时 0.3-0.5 秒。v4 的 NL 绿色按钮在固定位置、视觉突出、不受 IME 影响，命中率接近 100%，耗时 0.1-0.2 秒。

每天节省的时间：50 次 x (0.4 - 0.15) 秒 = 12.5 秒。时间不多，但节省的**认知负担**才是关键——每次"找 Return 键"都是一次注意力中断，而 NL 按钮把这个操作降级为无意识的肌肉记忆。

**痛苦解决率：** 80%。NL 按钮解决了"发送指令"的痛苦，但没有解决"换行但不发送"的痛苦（在 CC 的多行输入模式下需要先输入一行、换行、再输入下一行、最后发送）。这需要区分 NL（发送/Enter）和 Shift+Enter（换行不发送），v4 目前只有前者。不过从实际使用来看，Claude Code 的多行输入主要通过 `\` 续行符实现，纯粹的"换行不发送"需求相对低频。

### 网络感知 + 重连对信任的提升？

信任提升可以分成三个层次：

1. **v3 的信任基线：** "我知道 WebSocket 的状态"（四态指示器）
2. **v4 新增的信任层：** "我知道底层网络的状态"（NO NET 标签）+ "我知道 App 会自动修复"（NWPathMonitor + scenePhase 重连）
3. **最终信任感：** "我不需要关心网络，App 自己会搞定"

v3 到 v4 的信任提升主要在第 2 层到第 3 层的跨越。v3 让用户成为了一个"知情的观察者"——你知道状态，但需要自己判断是否要手动重连。v4 让用户成为了一个"被照顾的乘客"——App 在后台检测网络变化、在前台切换时检查连接、在网络恢复时自动重连，用户只需要使用，不需要管理。

**信任量化：** 如果用 0-10 衡量"使用 App 时对网络连接的焦虑程度"（10 = 极度焦虑），v2 大约是 7（经常看到假绿灯），v3 大约是 3（指示器透明但仍需手动管理），v4 大约是 1（几乎不需要想网络的事）。

### 零操作启动体验？

`autoConnectIfPossible()` 的实现简单直接：检查 `UserDefaults` 缓存的 `last_session_name` -> 在已加载的 session 列表中查找 -> 找到就 `selectedSession = session` 触发 fullScreenCover。没有复杂的缓存过期策略、没有预加载优化、没有启动动画——就是最朴素的"如果上次用的 session 还在，直接进去"。

这个简单性是优点。不需要处理 token 过期、不需要缓存终端内容、不需要 background fetch。唯一的前提是 session 还活着——如果 last session 已经被关闭了，`sessions.first(where:)` 返回 nil，用户正常看到 session 列表页面，没有任何副作用。

**与 Web 端对比：** Web 端打开 URL 后需要 (1) 加载 HTML + xterm.js CDN ~2s -> (2) 手动点 session -> (3) WebSocket 连接 ~0.5s。App 打开后 (1) 原生 UI 瞬间渲染 -> (2) fetch sessions + auto-connect ~1-2s。App 省掉了 CDN 加载和手动点击，总耗时少了 3-4 秒。

## 最想要的 5 个改进（v5 应该做什么）

### 1. 真机验证触摸滚动并修复问题
连续三个版本未验证的滚动功能已经成为最大的功能缺口。CC 的回复越来越长，不能可靠地回看历史严重限制了 App 的实用性。v5 必须在真机上跑通完整的滚动流程：手指滑动 -> tmux copy-mode -> 看历史 -> 惯性滚动 -> 点击退出。如果有 bug，立即修复。另外需要加一个浮动的"Exit Scroll"按钮——点击退出 copy-mode 的操作可发现性太差。

### 2. 动态字号适配
三个版本没动的终端可读性问题。iPhone SE/mini 上 12pt 字号可能只有 55-60 cols，很多 CC 的输出会折行变得不可读。Web 端的"cols < 70 时自动缩小字号"逻辑需要移植到 App 端。实现路径：SwiftTerm 的 `sizeChanged` 回调中检查 `newCols`，不足 70 时自动调小 `UIFont` 直到 cols >= 70 或字号到 8pt 下限。

### 3. 断线时终端区域半透明遮罩
当 WebSocket 断开时，终端显示的内容是最后一次同步的状态，可能已经过时了。用户看到了但不知道是不是最新的，会产生误判。加一个半透明黑色遮罩 + "Reconnecting..." 文字，明确告诉用户"看到的内容可能不是最新的，等重连后自动刷新"。

### 4. 快捷栏加 Select 按钮 + 改善复制体验
选择/复制是高频操作但可发现性极差。加 Select 按钮到快捷栏（和 Web 端一致），按下后进入选择模式。overlay 不默认全选，让用户自己拖选。加一个明确的"Copy"按钮在 overlay 上方。

### 5. 首次启动引导
现在 App 的"首次体验"是：打开 -> 看到一个错误页面（因为默认 IP 连不上）-> 用户不知道怎么办。需要一个简单的 onboarding：(1) 确保 Tailscale 已连接；(2) 在 Mac 上运行 start-claude.sh；(3) 输入 Mac 的 Tailscale IP。或者更好：支持 Bonjour/mDNS 自动发现同网段的 server.js，零配置即用。

## 会不会推荐给朋友？

直接推荐。

不再需要附加条件了。v3 的"如果你主要竖屏使用、打字以英文为主..."前提已经不需要了——v4 的中文输入方案经过了架构验证，NL 按钮解决了最高频操作，网络感知让通勤场景无忧。唯一的弱点是滚动回看（"如果你经常需要看 CC 的长回复，可能偶尔会遇到问题"），但这不是一个阻断性问题——大部分时候最新的终端输出就够了。

从 v2 的"不推荐" -> v3 的"可以有条件推荐" -> v4 的"直接推荐"，App 用了三个版本完成了从"能跑"到"好用"的蜕变。v4 证明了一件事：当核心交互（NL 按钮）、基础设施（网络感知 + 重连）、启动体验（auto-connect）三者同时做对的时候，一分之差也能带来质变。
