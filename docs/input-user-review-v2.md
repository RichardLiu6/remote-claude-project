# Web Terminal 移动端输入体验评测 v2

> 评测对象：`public/index.html` 中的移动端输入系统（v2 改动后）
>
> 评测环境：iPhone Safari，iOS 英文键盘（联想/自动纠正开启），搜狗/原生拼音输入法
>
> 评测日期：2026-03-07
>
> 基线对比：`docs/input-user-review-v1.md`（综合 6.7/10）

---

## 0. v2 变更清单

在打分之前先摸清 v2 到底改了什么。通过 `git diff HEAD~3 -- public/index.html` 确认，v2 的实际代码改动如下：

| # | 改动 | 具体内容 |
|---|------|----------|
| 1 | textarea 新增 `inputmode="text"` | 强制文本输入模式，理论上抑制某些 iOS 键盘变体 |
| 2 | NL 按钮 -> Enter 按钮 | 标签从 `"NL"` 改为 `"\u23CE"`（即 ⏎ 符号），发送序列从 `\x1b\r` 改为 `\r` |
| 3 | 新增 6 个 Ctrl 组合键按钮 | ^D / ^Z / ^A / ^E / ^L / ^R |
| 4 | 方向键/Tab 标签 Unicode 化 | `←→↑↓` 和 `⇧Tab` 用 Unicode 符号代替原来的纯文本 |

**没改的**（对照 v1 三大痛点）：

| v1 痛点 | v2 是否修复 | 说明 |
|---------|------------|------|
| 软键盘 Enter 被禁用 | **未修复** | `beforeinput` 仍然 `preventDefault()` 所有 `insertParagraph`/`insertLineBreak`，第 1225-1232 行不变 |
| Tab 补全后 sentBuffer 不同步 | **未修复** | `sendKey()` 仅 `wsSend(key) + focus()`，无 `resetSentState()` |
| compositionend 发送全量 textarea | **未修复** | 第 1184 行仍是 `const text = overlayInput.value; if (text) wsSend(text);`，未改为 `e.data` |

**注意**：项目 CLAUDE.md 描述的输入架构（"compositionend: records timestamp only; does NOT send text"、"beforeinput: post-IME 300ms suppression"）与实际代码**不一致**。CLAUDE.md 描述的是目标架构，代码尚未实现。

## 1. 十项评分（v1 vs v2 对比）

### 1.1 英文输入流畅度 — 7/10（v1: 7/10，不变）

diff 模型核心逻辑零改动。`sentBuffer` + 800ms `resetTimer` 的工作方式完全一样。新增的 `inputmode="text"` 理论上能防止 iOS 在某些场景下弹出数字/URL 键盘，但对英文输入流畅度本身无影响。

逐字输入、联想补全追加、快速连续打字——行为和 v1 完全一致，没有提升也没有退步。

### 1.2 中文输入体验 — 6/10（v1: 6/10，不变）

`compositionstart/compositionend` 处理逻辑**零改动**。v1 指出的三个问题全部保留：

1. **compositionend 仍然发送 `overlayInput.value` 而非 `e.data`**——如果 textarea 残留了之前 diff 发过的英文字符，compositionend 会重复发送。
2. **输入法切换时的边界状态**——sentBuffer 残留 + compositionend 全量发送 = 潜在重复。
3. **候选词联想触发 `input` 而非 composition**——diff 模型兜底，偶有闪烁。

v1 建议的修复方案（compositionend 中只发送 `e.data`）未被采纳。分数不变。

### 1.3 特殊键响应 — 6/10（v1: 5/10，+1）

**提分原因**：

1. Enter 按钮从 "NL" 改为 "⏎"，语义立刻清晰。用户看到 ⏎ 就知道这是回车，不用猜 "NL" 是什么意思。同时发送序列从 `\x1b\r`（Escape + CR）改为纯 `\r`，语义更正确——`\x1b\r` 在某些 shell/程序里可能被解释为 Alt+Enter 而非普通回车。
2. 新增 ^D / ^Z / ^A / ^E / ^L / ^R 六个 Ctrl 组合键按钮，覆盖了常用的终端控制操作（退出、挂起、行首、行尾、清屏、反向搜索）。v1 只有 ^C 一个 Ctrl 键，缺少这些操作要么做不到，要么需要外接蓝牙键盘。

**核心问题仍在**：软键盘 Enter 依然被完全禁用。`beforeinput` 里的 `insertParagraph`/`insertLineBreak` 仍然 `preventDefault()` + `resetSentState()`。用户按键盘上的 Return 键，什么都不发生。每条命令仍然需要"打字 + 找 ⏎ 按钮"两步操作。

这个问题的严重性值得反复强调：**它是整个输入系统最大的体验瓶颈**。v2 把 NL 改名为 ⏎ 降低了认知成本，但操作路径没有缩短——手指仍然要从键盘区域移动到快捷栏。如果能在非 composition 状态下恢复软键盘 Enter，这一项可以直接到 8 分。

只加 1 分而非更多，因为核心问题未解。

### 1.4 粘贴体验 — 7/10（v1: 7/10，不变）

零改动。Paste 按钮逻辑、iOS 原生粘贴走 diff 模型——全部不变。

### 1.5 输入框可见性 — 6/10（v1: 6/10，不变）

零改动。输入框仍然是"中转站"定位，800ms resetSentState 清空时闪烁，用户仍需适应"看终端不看输入框"的心智模型。`inputmode="text"` 不影响可见性。

### 1.6 快捷键栏 — 7/10（v1: 7/10，不变）

**改善的方面**：

1. ⏎ 比 NL 直觉得多，这是个好改动。
2. ^D/^Z/^A/^E/^L/^R 六个新按钮功能价值很高。能 ^D 退出 Python REPL、^L 清屏、^A/^E 跳行首行尾、^R 反向搜索历史——这些是每天都会用到的操作。
3. 方向键用 Unicode 箭头（←→↑↓）比纯文本更易识别。

**恶化的方面**：

按钮总数从 15 个膨胀到 **21 个**。原本一屏就放不下要滑动，现在更挤了。高频操作 Paste 和 Upload 被推到更远的位置，需要更多滑动才能触及。

快捷栏的根本矛盾加剧了：**功能越全越好 vs 单行放不下太多按钮**。6 个新 Ctrl 键很实用，但快捷栏的交互成本也水涨船高。考虑分组/折叠/长按弹出面板等方案可能比继续横向堆叠更合适。

功能增加和拥挤加剧相互抵消，分数不变。

### 1.7 键盘与滚动冲突 — 8/10（v1: 8/10，不变）

触摸滚动、`_keyboardOpen` re-focus、CSS 防弹簧、copy-mode 进出——全部不变，仍然是处理得最好的部分。

### 1.8 输入延迟感 — 7/10（v1: 7/10，不变）

输入链路零改动。diff 模型计算开销、WebSocket 往返延迟、iOS Safari 事件处理速度——全部和 v1 一致。

### 1.9 自动纠正干扰 — 8/10（v1: 8/10，不变）

textarea 原本已有 `autocomplete="off" autocorrect="off" autocapitalize="off" spellcheck="false"`。v2 新增了 `inputmode="text"`。

`inputmode="text"` 的作用是告诉浏览器弹出标准文本键盘（而非 URL/email/电话键盘）。对自动纠正行为本身影响甚微——iOS 的自动纠正是由 `autocorrect` 属性控制的，不是 `inputmode`。所以这项改动对自动纠正干扰几乎没有实际效果。

分数不变。

**关于"关闭 autocorrect 后打字感觉有何变化"**：v1 就已经关闭了 `autocorrect`，v2 没有额外的改变。实际体验中，iOS 部分版本仍不完全尊重 `autocorrect="off"`，联想栏（QuickType bar）有时仍会出现。`inputmode="text"` 不改变这个现状。

### 1.10 整体打字满意度 — 6/10（v1: 6/10，不变）

v2 的改动是"锦上添花"而非"解决痛点"。⏎ 按钮命名改善了认知体验，6 个 Ctrl 键扩展了功能覆盖面——但这些不触及核心问题。

用户的核心体验链路是："打字 -> 按回车执行"。这条链路在 v2 中仍然断裂为"打字 -> 移动手指到快捷栏 -> 找到 ⏎ 按钮 -> 点击"。只要这个断裂存在，整体满意度就上不去。

## 2. 真实场景模拟（v1 六场景复测 + 三个新场景）

### 场景 A：输入 "git push origin main" + Enter（复测）

**v1 表现**：打字流畅，Enter 被阻断，需要找 NL 按钮。
**v2 变化**：
1. NL 按钮变成了 ⏎ 按钮——更容易识别，视觉搜索时间缩短约 0.5 秒。
2. ⏎ 发送 `\r` 而非 `\x1b\r`——纯回车，语义正确。
3. 但操作路径不变：打字完毕 -> 眼睛从键盘移到快捷栏 -> 手指点 ⏎。

**用户感受**：比 v1 好一点——至少不用猜 NL 是什么了，⏎ 一目了然。但流程仍然是两步操作。改善幅度：从"恼火"降级为"无奈"。

### 场景 B：输入中文 "修复登录页面的样式问题"（复测）

**v2 变化**：零。compositionend 仍发 `overlayInput.value`，每次 compositionend 后 resetSentState 清空 textarea 仍会闪。

**用户感受**：与 v1 完全一致。

### 场景 C：输入 "th" 后点击联想建议 "the"（复测）

**v2 变化**：零。diff 模型不变，追加型补全仍然完美处理。

### 场景 D：快速输入 "npm install express ws node-pty"（复测）

**v2 变化**：零。快速打字体验不变，打完后仍要找 ⏎ 按钮。

### 场景 E：粘贴一个 URL（复测）

**v2 变化**：零。两种粘贴方式行为不变。

### 场景 F：按 Tab 触发补全，然后继续输入（复测）

**v2 变化**：零。sentBuffer 在 Tab 后仍不同步。输入 "serv" -> Tab -> shell 补全为 "server.js" -> sentBuffer 仍是 "serv" -> 继续输入会基于错误的 sentBuffer 计算 diff。

这是 v1 指出的第二大痛点，修复方案很简单（Tab 发送后立即 `resetSentState()`），但 v2 未实施。

### 场景 G（新）：按 ^D 退出

**操作过程**：
1. 在 Python REPL 或 cat 等待输入的状态下
2. 滑动快捷栏找到 ^D 按钮 -> 点击
3. `sendKey('\x04')` -> wsSend -> 远端收到 EOF -> 程序退出

**用户感受**：功能正确。^D 按钮发送 `\x04`（EOT），这是标准的终端 EOF 信号。Python REPL 收到后正常退出，cat 收到后停止读取。v1 做不到这个操作（除非外接蓝牙键盘），现在可以了。

**问题**：^D 按钮在快捷栏的第 9 个位置（⏎, Tab, ⇧Tab, ←, →, ↑, ↓, ^C, **^D**），需要滑动一小段才能看到。如果用户不知道快捷栏可以横向滑动，可能找不到这个按钮。但对于知道 Ctrl+D 的用户来说，^D 标签足够直觉。

### 场景 H（新）：按 ^L 清屏后继续输入

**操作过程**：
1. 终端上有大量输出，想清屏
2. 滑动快捷栏找到 ^L 按钮（第 13 个位置）-> 点击
3. `sendKey('\x0c')` -> wsSend -> shell 收到 SIGWINCH/clear -> 终端刷新
4. 继续输入新命令

**用户感受**：清屏功能正确。但 ^L 在快捷栏非常靠后，要滑动很远。从用户视角，清屏是个中频操作（每天可能用几十次），但要滑两三下才能找到按钮。

**关键问题**：^L 清屏后，sentBuffer 和 overlayInput.value 的状态不受影响（因为清屏是服务端行为，不涉及客户端输入状态）。如果清屏前正在打字，清屏后 sentBuffer 仍保留之前的内容，继续打字不会出错——diff 模型从上次的 sentBuffer 继续追加。这一点处理正确。

### 场景 I（新）：快速输入后立即按 Enter

**操作过程**：
1. 快速输入 "ls -la"
2. 立即按键盘上的 Return 键
3. `beforeinput` 拦截 `insertParagraph` -> `preventDefault()` -> `resetSentState()`
4. "ls -la" 已经通过 diff 模型发送到终端了，但回车没有发送
5. 手指移到快捷栏 -> 点 ⏎ -> 发送 `\r`

**用户感受**：这是最典型的"Enter 被禁用"痛点场景。用户的肌肉记忆是"打完命令立刻按 Return"，整个动作应该是一气呵成的。结果 Return 被吞了，还要额外点一次 ⏎。

**额外的坑**：步骤 3 中 `resetSentState()` 清空了 sentBuffer 和 overlayInput.value。如果用户在按 Return 和按 ⏎ 之间继续打字（比如突然想改命令），新输入会基于空的 sentBuffer 开始——这没问题，但如果用户看到输入框闪了一下空白可能会困惑。

## 3. 新增 Ctrl 组合键专项评价

### 3.1 好用的地方

| 按键 | 功能 | 实用频率 | 评价 |
|------|------|----------|------|
| ^D | EOF/退出 | 高 | 退出 Python、node REPL 的唯一便捷方式，必备 |
| ^C | 中断 | 极高 | v1 已有，不变 |
| ^A | 行首 | 高 | 编辑长命令时跳到行首，非常实用 |
| ^E | 行尾 | 高 | 配合 ^A 使用，编辑命令必备 |
| ^L | 清屏 | 中 | 比输入 "clear" 快，但位置太靠后 |
| ^R | 反向搜索 | 中 | 搜索历史命令，高级用户常用 |
| ^Z | 挂起 | 低 | 挂起前台进程，使用频率较低 |

### 3.2 不好用的地方

1. **位置过于分散**。7 个 Ctrl 键（含原有 ^C）散布在 21 个按钮中，没有视觉分组。^C 在第 8 个位置，^D 在第 9 个，^Z 在第 10 个，^A 第 11 个，^E 第 12 个，^L 第 13 个，^R 第 14 个——中间被方向键和 ^C 隔开。用户要"扫描"才能找到目标。

2. **缺少 Ctrl+W（删除前一个单词）**。这是 shell 编辑中仅次于 Backspace 的高频删除操作。打错一个长路径时，^W 比连按 Backspace 快得多。

3. **缺少 Ctrl+U（删除到行首）**。和 ^W 一样是高频编辑操作。打了一长串命令想全部删掉重来，^U 一键清空，比按住 Backspace 等它慢慢删快。

4. **没有利用物理键盘的 Ctrl 键**。如果用户外接蓝牙键盘，keydown 里没有 `e.ctrlKey` 的处理逻辑——Ctrl+D 等组合键在蓝牙键盘上不工作（只能通过快捷栏按钮）。这是一个疏漏。

### 3.3 建议

- 将 Ctrl 键分组：常用编辑（^A ^E ^W ^U）| 信号（^C ^D ^Z）| 工具（^L ^R），视觉上用分隔符或不同底色区分
- keydown 增加 `e.ctrlKey` 检测，让蓝牙键盘也能用 Ctrl 组合键
- 补充 ^W 和 ^U

## 5. 总结

### v2 做对了什么

1. **⏎ 按钮命名**——从 "NL" 到 "⏎"，认知成本显著下降。这是成本最低、效果最明显的改动。
2. **发送序列修正**——`\r` 比 `\x1b\r` 语义正确。`\x1b\r` 可能在某些程序中被误解析为 Alt+Enter。
3. **Ctrl 组合键覆盖面**——从 1 个（^C）扩展到 7 个，覆盖了终端操作的大部分常用 Ctrl 键。对高级用户来说，^A/^E/^D/^L 是每天要用的。

### v2 没做什么（需要在 v3 解决的）

| 优先级 | 问题 | v1 就提过 | 影响 |
|--------|------|-----------|------|
| P0 | 软键盘 Enter 仍被禁用 | 是（痛点 #1） | 每条命令多一步操作 |
| P1 | Tab 补全后 sentBuffer 不同步 | 是（痛点 #2） | 补全后继续输入可能乱码 |
| P1 | compositionend 发送全量 textarea | 是（痛点 #3） | 输入法切换边界重复发送 |
| P2 | 快捷栏 21 按钮过于拥挤 | 是（v1 说 15 个就多了） | 高频按钮需要滑动 |
| P2 | 缺少 ^W / ^U | 新发现 | 编辑长命令效率低 |
| P2 | 蓝牙键盘 Ctrl 组合键不工作 | 新发现 | 外接键盘用户体验断裂 |
| P3 | CLAUDE.md 与代码不一致 | 新发现 | 维护混乱，下一个开发者会被误导 |

### 对工程师的话

v2 的改动方向是对的——改善命名、扩展功能覆盖面。但三个核心痛点一个都没碰。从 6.7 到 6.8，提升幅度几乎在误差范围内。

如果只能做一件事，请做这件：**在非 composition 状态下恢复软键盘 Enter**。具体方案：`beforeinput` 里检测 `insertParagraph`/`insertLineBreak` 时，如果 `isComposing === false` 且距离上一次 `compositionend` 已超过 300ms，就发送 `\r` 而非 `preventDefault()`。这一个改动可以把特殊键响应从 6 提到 8，整体满意度从 6 提到 7.5，综合分从 6.8 提到 7.5 以上。

Enter 键是终端输入的灵魂。没有它，其他一切优化都是隔靴搔痒。
